# Plan: Difficulty Tiers & Doc-Sourced Questions for Coderegon Trail

## Context

The current games have 6 quiz events each, all shown to every player regardless of chosen profession. The `difficulty` field exists in trail data but the game code ignores it. Questions lean toward engineer-level (38%) with decent vibe coder coverage (45%) but sparse expert questions (18%). Major framework concepts from official docs are missing (e.g., Django admin/signals, Rails associations/scopes, React hooks rules/keys, Laravel facades/queues).

The user wants:
1. Questions sourced from framework docs at appropriate skill levels
2. Easier questions for vibe coders, harder for engineers
3. A mechanism for both easier and harder game versions

## Approach: Tiered Question Pool + Profession-Based Selection

### 1. Expand events in `framework-trails.md` (12-15 per framework, up from 6)

Each framework gets questions at 3 tiers sourced from official docs:

| Tier | Tag | Who sees it | Question style |
|------|-----|-------------|----------------|
| `easy` | Vibe Coder + Ralph | "What does X do?" / "Which is the right pattern?" |
| `medium` | Vibe Coder + Engineer | "Why does X work this way?" / "What happens if...?" |
| `hard` | Engineer + Staff Architect | "Spot the bug" / "Trace the flow" / "What's the tradeoff?" |

**Per-framework concept additions** (from doc research):

| Framework | Easy additions | Medium additions | Hard additions |
|-----------|---------------|-----------------|----------------|
| **Django** | MVT pattern, admin site, migrations | signals, form validation, Q objects | lazy evaluation, connection pooling, raw SQL |
| **Rails** | RESTful resources, generators, associations | scopes, callbacks, nested routes | metaprogramming, callback order, thread safety |
| **Express** | static files, route params, Router | error-handling middleware (4 args), next() | backpressure, event loop blocking, async context |
| **React** | props vs state, JSX, lists+keys | hooks rules, useEffect cleanup, controlled forms | reconciliation, stale closures, batching |
| **Laravel** | Blade syntax, seeders, relationships | scopes, accessors, request validation | service providers, queue jobs, facades internals |

### 2. Update game code to filter events by profession

```
Profession mapping:
  Ralph Wiggum  → sees: easy only (+ forgiving mode)
  Vibe Coder    → sees: easy + medium
  Engineer      → sees: medium + hard
  Staff Architect → sees: hard only (+ no hints)
```

In each game's HTML, modify the event selection logic:
- Current: `events[eventIndex]` (sequential)
- New: filter `TRAIL_DATA.events` by `difficulty` matching profession tier, then pick sequentially from filtered list
- Each game still shows ~6 events per playthrough, drawn from the appropriate tier

### 3. Show difficulty badge in event UI

Add a small tag in the event overlay showing the tier: `[EASY]`, `[MEDIUM]`, `[HARD]` — styled like the existing tag badges on the landing page.
